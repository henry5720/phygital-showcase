# AR V2 官網互動驗證頁設計文件

**日期**: 2026-06-06  
**狀態**: 已確認，待 spec review

## 1. 目標

建立一個獨立的 `/ar/v2` 驗證頁，先使用 MindAR 官網 interactive demo 的官方素材與互動模式，確認以下能力可在目前專案中正常運作：

- 相機啟動與 MindAR image target 掃描
- 掃描成功後顯示 3D model 與 2D panel
- A-Frame clickable 互動
- preview 影片播放
- 離開頁面後的相機與 overlay 清理

這一版是技術驗證頁，不是正式產品體驗頁。

## 2. 非目標

- 不替換成品牌自有 logo、影片、圖片或文案
- 不修改現有 `/ar` 與 `/ar/scanner` 流程
- 不將 A-Frame 寫法抽象成長期共用 AR 架構
- 不把官網互動完整 1:1 還原到所有動畫與導外連結

## 3. 路由策略

保留現有正式流程：

- `/ar`
- `/ar/scanner`

新增獨立驗證路由：

- `/ar/v2`

`/ar/v2` 與現有 Three.js 版完全隔離，避免驗證過程干擾已存在的正式流程。

## 4. 技術選擇

### 推薦方案

採用 `Page + Scene` 拆層：

- `ArV2Page`: React route page，負責頁面殼層與掛載點
- `ArV2Scene`: 承接 A-Frame `<a-scene>`、`<a-assets>`、MindAR target entity
- `ar-v2` 初始化模組: 處理 target found、內容顯示順序、clickable 互動、影片播放

### 不採用方案

1. 單頁完整移植 `advanced.html`
原因: 最快，但全域 script 與 HTML 耦合過重，之後難以維護。

2. 直接改寫成現有 `MindARThree + React`
原因: 現階段目標是先驗證能跑，不是重做互動與架構。

3. 只用靜態 HTML 或 iframe 載入官網 sample
原因: 驗證不到目前專案的實際整合能力。

## 5. 檔案與責任分工

建議檔案結構：

```text
src/
├── pages/
│   └── ArV2Page.tsx
├── components/
│   └── ArV2Scene.tsx
└── lib/
    └── ar-v2.ts

public/
└── assets/
    └── ar-v2/
        ├── targets/
        ├── icons/
        ├── portfolio/
        ├── models/
        └── videos/
```

責任切分：

- `ArV2Page.tsx`
  - 路由入口
  - 顯示簡單說明、返回按鈕、必要 debug 訊息
  - 掛載 `ArV2Scene`

- `ArV2Scene.tsx`
  - 渲染 A-Frame scene 與 assets
  - 建立 target entity 與可點擊元素
  - 管理掛載與卸載時的初始化、清理

- `src/lib/ar-v2.ts`
  - 封裝官網 demo 的 imperative 互動邏輯
  - 包含 avatar 進場、portfolio 顯示、按鈕切換、preview 播片

## 6. 素材策略

`v2` 所需素材全部本地化，不依賴外部 CDN。

建議放在 `public/assets/ar-v2/`，與未來正式品牌素材分離。

本地化的理由：

- 避免第三方 CDN 失效或版本變動造成誤判
- 降低 CORS 與網路延遲對驗證結果的干擾
- 後續替換成品牌素材時，路徑結構更清楚

## 7. 第一版互動範圍

### 保留的能力

1. 掃描 target 成功後觸發內容顯示
2. 顯示 avatar / 3D model
3. 顯示 portfolio panel
4. portfolio 左右切換
5. 點 preview 播放至少一支影片

### 暫不保留的能力

1. 所有 profile / web / email / location 資訊 icon 完整複製
2. 官網所有動畫 timing 與細節 1:1 還原
3. 導外連結行為

這樣的範圍足以驗證「官網互動模式能在專案內跑起來」，但不會過早把驗證頁膨脹成產品功能。

## 8. 互動流程

```text
進入 /ar/v2
  -> A-Frame scene 初始化
  -> 啟動相機與 MindAR 掃描
  -> 掃描到官方 target
  -> 觸發 targetFound
  -> 顯示 avatar / 3D model
  -> 顯示 portfolio panel
  -> 使用者可左右切換 panel item
  -> 使用者可點擊 preview 播放影片
  -> 離開頁面時清理 scene、overlay、相機資源
```

## 9. 驗證標準

`v2` 完成條件定義如下：

1. `/ar/v2` 可正常進入
2. 相機可啟動，沒有立即報錯
3. 掃描官方 target 後內容穩定出現
4. 3D model 與 portfolio panel 可見
5. portfolio 切換與影片播放至少各成功一次
6. 離開頁面後沒有殘留相機或掃描 overlay 問題

## 10. 測試策略

### 靜態整合驗證

- Router 已掛上 `/ar/v2`
- `ArV2Page` 能渲染 `ArV2Scene`
- scene 中存在必要的 assets 與 target entity
- 所有素材皆指向本地路徑

### 執行期驗證

- 相機權限流程正常
- `targetFound` 事件可觸發
- avatar / panel / preview 依序出現
- panel item 可切換
- 影片可開始播放

### 清理驗證

- 離開頁面後不殘留相機占用
- 不殘留 MindAR scanning overlay
- 切回其他 route 不受 A-Frame 全域狀態污染

## 11. 風險與邊界

1. A-Frame 與現有 Three.js 版共存
風險控制: `v2` 使用獨立 route 與獨立元件，不與既有 `MindArCanvas.tsx` 共享生命週期。

2. 手機瀏覽器影片播放限制
風險控制: 第一版只驗證點擊後可播放，不依賴 autoplay。

3. MindAR overlay 與相機資源未完全清理
風險控制: 將清理列為正式驗收條件，而不是附帶項目。

## 12. 後續延伸方向

若 `/ar/v2` 驗證成功，下一步再決定以下其一：

1. 繼續保留 A-Frame 版作為正式體驗基礎
2. 只保留互動結構，改寫回現有 `MindARThree + React`
3. 用 `v2` 驗證結果挑出最值得保留的互動，再導入品牌素材

目前不在此文件中預先決定最終正式架構，避免在驗證前做過度承諾。
